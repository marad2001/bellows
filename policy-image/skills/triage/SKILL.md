# Triage

Read one issue carefully. Pick a state. Either route it to an implementer (agent or human), request more info from the reporter, or close it with a precedent.

This skill is the source of truth for the role taxonomy, the brief templates, the AI-disclaimer wording, and the decision heuristics. Bellows's containerised flow (`bellows triage`) consumes this skill in headless mode — the per-invocation kickoff prompt at `/workspace/.bellows-kickoff.md` layers a headless-override on top (no `gh`, no human-wait, write a JSON verdict to a file instead of mutating GitHub directly). When invoked manually with the `gh` CLI available, this skill drives the GitHub-side mutations directly.

## Role taxonomy

Every triage verdict picks **exactly one state** for the issue:

| State              | Meaning                                                              | Outcome                                              |
| ------------------ | -------------------------------------------------------------------- | ---------------------------------------------------- |
| `needs-info`       | Too vague to action; reporter needs to answer specific questions.    | Post a comment listing the questions.                |
| `ready-for-agent`  | Bounded, fully specified; an AFK Claude Code agent can implement it. | Post an `## Agent Brief` comment; relabel.           |
| `ready-for-human`  | Requires human judgement (architecture, design, cross-context).      | Post a `## Human Brief` comment; relabel.            |
| `wontfix`          | Will not be actioned.                                                | Close the issue; for enhancements, record precedent. |

And **exactly one category**:

| Category      | Meaning                                                                       |
| ------------- | ----------------------------------------------------------------------------- |
| `bug`         | Something is broken. The issue describes the breakage and (ideally) a repro.  |
| `enhancement` | Something is missing or could be better. Not a regression — a new ask.        |

Category matters in two places: (a) it sets the GitHub `bug` / `enhancement` label, and (b) `wontfix` + `enhancement` triggers the out-of-scope precedent flow (see below).

The canonical label strings live in `docs/agents/triage-labels.md`. Treat them as authoritative — bellows keys label transitions on these exact strings.

## AI disclaimer

Every comment posted as part of a triage verdict is prefixed with this exact line on its own line, followed by a blank line, then the rest of the body:

```
> *This was generated by AI during triage.*
```

A human reading the issue must be able to tell at a glance that the verdict came from triage. Do not paraphrase the disclaimer.

## Decision tree

Walk these checks in order. Stop at the first one that fires.

1. **Is the issue too vague to action?**
   - No repro for a bug, "make it better" with no concrete ask for an enhancement, missing acceptance criteria.
   - → `needs-info`. Write the specific questions you need answered in `comment_body`. 1–3 sharply scoped questions. Avoid stacked dependent questions ("first tell me X, then if X is Y, tell me Z") — ask the first question, re-triage when the answer arrives.

2. **Does the issue conflict with an established `.out-of-scope/` precedent or an explicit ADR rejection?**
   - Read `.out-of-scope/` and `docs/adr/` before deciding. A duplicate of a wontfix precedent is itself wontfix.
   - → `wontfix`. For an enhancement, write a new `.out-of-scope/<slug>.md` capturing the new precedent (with the originating issue link so future triage can trace it). For a bug, just close.

3. **Is the issue bounded, well-specified, and one-PR-shaped?**
   - The acceptance criteria can be enumerated. The change touches one bounded area. An agent can plausibly finish it inside a single PR.
   - → `ready-for-agent`. Write an `## Agent Brief` carefully — the downstream agent treats it as the contract.

4. **Does the issue require human judgement?**
   - Architectural calls, multi-area changes, decisions between conflicting ADRs, security-sensitive changes, cross-context migrations.
   - → `ready-for-human`. Write a `## Human Brief` for the human implementer.

When in genuine doubt, prefer `needs-info` — better a clean re-triage on a future tick than an applied verdict that's wrong.

### needs-info vs grill

For very-vague enhancements two routes exist:

- **`needs-info`** — ask 1–3 specific questions, label the issue, wait for the reporter. Use when the gap is a small number of concrete pieces of information the reporter can supply.
- **Grilling** (a separate skill at `~/.claude/skills/grill-with-docs/`) — drag the reporter through structured back-and-forth until the request is well-shaped. Interactive; not available in bellows's headless flow.

In bellows's headless flow (`bellows triage`), grilling is out of scope — choose `needs-info` and capture the questions.

## Brief templates

### `## Agent Brief` — for `ready-for-agent`

A bellows-run agent reads this brief out of an issue comment via `tracker::fetch_agent_brief`, which scans for the literal `## Agent Brief` header. The downstream agent then treats the brief's `Acceptance criteria` checklist as the definition of done. Be specific.

```markdown
## Agent Brief

**Category:** bug | enhancement
**Summary:** One sentence — what changes and why.

**Current behavior:**
Plain-prose description of how the code behaves today (call out the specific files / functions / data flow involved).

**Desired behavior:**
Plain-prose description of how the code should behave after the change. Tie back to the user-visible outcome.

**Key interfaces:**
- `path/to/file.rs` — what changes here
- `path/to/other.rs` — and here

**Acceptance criteria:**
- [ ] Behaviour-1 test: short description.
- [ ] Behaviour-2 test: short description.
- [ ] `cargo test --all-targets --all-features` and `cargo clippy --all-targets --all-features -- -D warnings` both green.
- [ ] Manual smoke (if applicable; describe the steps).

**Out of scope:**
- Anything the downstream agent should not try to tackle in this PR.
```

### `## Human Brief` — for `ready-for-human`

```markdown
## Human Brief

**Why this needs a human:** One paragraph. Name the specific judgement an agent cannot make.

**Decisions to make:**
- Decision 1: options, trade-offs, what biases the call.
- Decision 2: ...

**Context:**
- Relevant ADRs / precedents.
- Prior discussion / related issues.

**Suggested next step:** What would unblock the human implementer (a spike, a sync, etc.)?
```

## `wontfix` + `enhancement`: record the precedent

When a triage verdict is `wontfix` on an `enhancement`, write a new `.out-of-scope/<slug>.md` file capturing the precedent:

```markdown
# <Title — the thing we won't do>

**Rejected because:** Plain-prose reasoning (constraint, scope, prior decision).

**Originating issue:** #<n> — <title>

**See also:** Optional links to related ADRs / precedents / discussion.
```

The file lands on `master` directly (no PR) so future triage runs can spot duplicates by scanning `.out-of-scope/`. The closing comment on the issue should link to the new precedent file so the reporter can find it.

For `wontfix` + `bug`, just close — no precedent file needed.

## Stop conditions

You are done when the issue has been routed to exactly one of the four states, the appropriate comments have been posted, the labels transition cleanly, and (for `wontfix` + `enhancement`) the new `.out-of-scope/` precedent is committed to master.

In bellows's headless flow, the agent stops at "structured JSON verdict written to `/workspace/.bellows-triage-verdict.json`" — bellows then drives the GitHub-side mutations on the host.
